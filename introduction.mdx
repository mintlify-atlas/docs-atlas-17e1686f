Meteor is a plugin-driven agent for collecting metadata. Written in Go, it compiles into a single binary with no external dependencies. You point it at a data source — a database, a dashboard, a message queue, a cloud service — and it extracts metadata and forwards it to any catalog or downstream system.

Metadata flows through three stages: an **extractor** pulls from a source, optional **processors** transform the data, and one or more **sinks** deliver it to its destination. You wire these together in a YAML file called a **recipe**.

## How it works

Every extraction job is described by a recipe — a YAML file with three sections:

- **`source`** — the extractor plugin and its configuration (credentials, connection strings, scope)
- **`processors`** — optional plugins that enrich or transform metadata before it leaves
- **`sinks`** — one or more destination plugins that receive the final metadata

A minimal recipe looks like this:

```yaml
name: my-postgres-job
version: v1beta1
source:
  name: postgres
  scope: production
  config:
    connection_url: "postgres://user:pass@localhost:5432/mydb"
sinks:
  - name: console
```

You run it with:

```bash
meteor run my-postgres-job.yaml
```

Meteor prints a run summary — recipe name, source, duration, and record count — for each job.
