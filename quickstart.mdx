---
title: "Quickstart"
description: "Install Meteor, write a recipe, and run your first metadata extraction in minutes."
---

This guide walks you through installing Meteor, creating a recipe that extracts metadata from a source, and running it to see output in your terminal.

<Info>
  This quickstart uses the `console` sink, which prints extracted metadata to standard output. No external service is required.
</Info>

<Steps>
  <Step title="Install Meteor">
    Install Meteor using Homebrew on macOS or Linux:

    ```bash
    brew install raystack/tap/meteor
    ```

    Verify the installation:

    ```bash
    meteor version
    ```

    For other installation methods — binary download, Docker, or building from source — see the [Installation](/installation) page.
  </Step>

  <Step title="List available plugins">
    Before writing a recipe, explore the plugins available on your installation. Plugins are grouped into extractors, processors, and sinks.

    ```bash
    # List all available extractors
    meteor list extractors

    # List all available sinks
    meteor list sinks

    # List all available processors
    meteor list processors
    ```

    To get details about a specific plugin — including its sample config and expected output — use the `info` command:

    ```bash
    meteor info extractor postgres
    meteor info sink console
    ```
  </Step>

  <Step title="Generate a recipe">
    Use the `meteor new` command to generate a starter recipe. The `-n` flag sets the URN namespace (required). Redirect stdout to save it to a file:

    ```bash
    meteor new recipe my-recipe -n my-org -e bigquery -s console > my-recipe.yaml
    ```

    Open `my-recipe.yaml` and fill in your source credentials:

    ```yaml
    name: my-recipe
    version: v1beta1
    source:
      name: bigquery
      scope: my-project          # URN namespace for this source
      config:
        project_id: my-gcp-project
        service_account_base64: "<base64-encoded-service-account-json>"
    sinks:
      - name: console
    ```

    <Tip>
      You can generate a recipe for any extractor. Run `meteor list extractors` to see what's available, then substitute the extractor name in the `-e` flag.
    </Tip>

    **Using environment variables in recipes**

    To avoid hardcoding credentials, use Go template syntax. Any environment variable prefixed with `METEOR_` is available as template data:

    ```yaml
    name: my-recipe
    version: v1beta1
    source:
      name: postgres
      scope: production
      config:
        connection_url: "{{ .connection_url }}"
    sinks:
      - name: console
    ```

    Then set the variable before running:

    ```bash
    export METEOR_CONNECTION_URL=postgres://user:pass@localhost:5432/mydb
    ```
  </Step>

  <Step title="Validate the recipe">
    Before running, lint your recipe to catch any formatting or configuration errors:

    ```bash
    meteor lint my-recipe.yaml
    ```

    If the recipe is valid, the command exits with no errors.
  </Step>

  <Step title="Run the recipe">
    Execute the recipe to extract metadata:

    ```bash
    meteor run my-recipe.yaml
    ```

    Meteor runs the extraction and prints a summary when it completes:

    ```text
    All recipes ran successful
    0 failing, 1 successful, and 1 total

    Status  Recipe     Source    Duration(ms)  Records
    ✓       my-recipe  bigquery  1240          42
    ```

    Extracted metadata is printed to the console by the `console` sink.

    <Note>
      To run all recipes in a directory, pass the directory path instead of a file:

      ```bash
      meteor run _recipes/
      ```
    </Note>
  </Step>
</Steps>
